# `problem.md` 작성 규칙

`problem.md`는 AOJ 스타일 문제 하나의 한국어 문제 지문입니다.

반드시 한국어로 작성합니다. 사용자가 다른 언어를 명시적으로 요청하지 않는 한, 제목, 문제 설명, 입력 설명, 출력 설명, 힌트는 모두 한국어로 씁니다.

문서는 Markdown 형식으로 작성하며, 수식은 KaTeX 문법을 사용합니다.

## 작성 순서

문제 생성 작업에서는 `problem.md`를 가장 먼저 작성합니다.

`validator.cpp`는 `problem.md`가 충분히 명확해진 뒤에만 작성합니다. 따라서 `problem.md`에는 validator가 필요한 입력 형식, 범위, 구조적 제약이 모두 들어 있어야 합니다.

작성 중 모호한 점이 있으면 임의로 채우지 말고 사용자에게 질문합니다. 특히 입력 형식, 제약 조건, 그래프/트리 조건, 예제 출력, 정답 정의가 불명확하면 반드시 질문합니다.

- 인라인 수식: `$a_i \le N$`
- 블록 수식:

```md
$$
\sum_{i=1}^{n} i = \frac{n(n+1)}{2}
$$
```

## 기본 서식

`problem.md`는 기본적으로 아래 구조를 따릅니다.

````md
# {문제 제목}

## 문제

문제 설명을 입력하세요.

## 입력

입력 형식을 설명하세요.

## 출력

출력 형식을 설명하세요.

## 예제 입력 1

```
예제 입력
```

## 예제 출력 1

```
예제 출력
```

## 힌트

LaTeX 수식 예시: $a^2 + b^2 = c^2$

블록 수식:
$$
\sum_{i=1}^{n} i = \frac{n(n+1)}{2}
$$
````

## 섹션 규칙

### `# {문제 제목}`

문제 제목을 적습니다.

사용자가 문제 제목을 준 경우 그 제목을 그대로 사용합니다. 제목을 정해야 한다면 짧고 자연스러운 한국어 제목을 사용합니다.

### `## 문제`

문제에서 주어지는 상황과 해야 할 일을 설명합니다.

반드시 포함할 내용:

- 무엇이 주어지는지
- 무엇을 구해야 하는지
- 답의 정확한 정의
- 필요한 경우 용어 정의

문제 설명은 한국어로 자연스럽게 작성하되, 조건은 모호하지 않게 씁니다.

### `## 입력`

입력 형식을 줄 단위로 정확히 설명합니다.

예:

```md
첫째 줄에 정수 $N$과 $M$이 공백으로 구분되어 주어진다.
둘째 줄에 $N$개의 정수 $a_1, a_2, \ldots, a_N$이 공백으로 구분되어 주어진다.
다음 $M$개의 줄 중 $i$번째 줄에는 두 정수 $u_i$, $v_i$가 공백으로 구분되어 주어진다.
```

각 변수에 대해 다음을 명확히 적습니다.

- 의미
- 범위
- 반복 횟수
- 인덱스가 0-based인지 1-based인지
- 줄바꿈과 공백 형식

### `## 출력`

출력해야 하는 값을 정확히 설명합니다.

실수 출력 문제라면 허용 오차 또는 필요한 출력 정밀도를 명시합니다.

### `## 예제 입력 {번호}`

예제 입력 번호는 1부터 시작하는 정수입니다. 즉 `{번호}`는 1-based입니다.

예제 입력은 반드시 아래 형식을 사용합니다.

````md
## 예제 입력 1

```
3
1 2 3
```
````

예제가 여러 개라면 번호를 1씩 증가시킵니다.

- `## 예제 입력 1`
- `## 예제 출력 1`
- `## 예제 입력 2`
- `## 예제 출력 2`

입력과 출력의 번호는 반드시 서로 일치해야 합니다.

### `## 예제 출력 {번호}`

예제 출력 번호는 대응하는 예제 입력 번호와 같습니다. 즉 `{번호}`는 1-based입니다.

예제 출력은 반드시 아래 형식을 사용합니다.

````md
## 예제 출력 1

```
6
```
````

### `## 힌트`

예제 설명, 풀이 힌트, 추가 설명을 적습니다.

힌트가 필요 없다면 짧게 생략해도 됩니다. 단, 사용자가 예제 설명을 요청했거나 예제가 헷갈릴 수 있다면 `## 힌트`에 예제의 답이 왜 그렇게 나오는지 설명합니다.

## 제약 조건 작성

제약 조건은 별도 섹션이 필요하면 `## 문제`, `## 입력`, 또는 `## 힌트` 안에 자연스럽게 포함하거나, 문제 형식상 필요할 경우 `## 제약` 섹션을 추가할 수 있습니다.

validator가 확인할 수 있을 만큼 모든 제약을 명확히 적습니다.

예:

```md
- $1 \le N \le 200000$
- $0 \le M \le 200000$
- $1 \le u_i, v_i \le N$
- $u_i \ne v_i$
- 주어지는 간선은 모두 서로 다르다.
- 그래프는 연결되어 있다.
```

구조적 제약도 빠뜨리지 않습니다.

- 그래프가 트리인지, DAG인지, 단순 그래프인지
- 그래프가 방향 그래프인지 무방향 그래프인지
- 배열이 순열인지, 정렬되어 있는지
- 문자열에 허용되는 문자가 무엇인지
- 격자에 허용되는 문자가 무엇인지
- 특정 문자가 정확히 몇 번 등장해야 하는지

## 추측 금지

아래 정보가 없으면 임의로 추측하지 말고 사용자에게 질문합니다.

- 테스트 케이스 개수
- 정확한 입력 형식
- 숫자 범위
- 인덱스가 0-based인지 1-based인지
- 그래프가 방향 그래프인지 무방향 그래프인지
- self-loop 허용 여부
- 중복 간선 허용 여부
- 그래프 연결성 조건
- 문자열 허용 문자
- 격자 허용 문자
- 배열의 정렬, 중복, 순열 조건
